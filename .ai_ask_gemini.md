# Gemini 리뷰 요청: dataflow-saas 파이프라인 설계

## 배경

이직 준비용 포트폴리오 프로젝트. 현 직장(B2B 데이터 분석 SaaS)에서 배운 파이프라인 패턴을 일반화해서 직접 구현하려고 한다.

**현재 상태:**
- ingestion/collector (GitHub API → JSON/Parquet) - 동작함
- transform/converter (Excel → Parquet) - 동작함
- storage/warehouse (DuckDB) - 동작함
- 각각 독립적으로 존재. 연결되어 있지 않음.

**목표:** 원본 데이터 → 최종 차트 데이터까지 전체 파이프라인 설계

## 파이프라인 설계 초안

회사 패턴(Convert→Tag→TagPrep→Load→SRCH→S→H→RPT)을 업계 표준(Medallion + dbt 패턴)으로 일반화한 8단계:

```
1. Collect    - 데이터 수집 (API → JSON)                    [Bronze]
2. Convert    - 구조 변환 (JSON → Parquet, 타입 변환)         [Bronze]
3. Enrich     - 메타데이터 추가 (source, tenant, ingested_at) [Bronze→Silver]
4. Load       - DuckDB 적재 (Parquet → 테이블)               [Silver]
── 여기까지 "데이터를 넣는" 단계 ──
5. Staging    - 필요한 컬럼만 추출 (View)                     [Silver]
6. Intermediate - 필터, 집계, JOIN                           [Silver]
7. Mart       - KPI 계산, 크로스 테이블 연산                   [Gold]
8. Report     - 차트용 포맷 변환                               [Gold]
── 여기까지 "데이터를 만드는" 단계 ──
→ FastAPI로 Report 테이블 서빙
```

## 기술 스택
- Python 3.11+, DuckDB, Airflow (Docker), FastAPI
- 로컬 환경 (클라우드 의존성 없음)

## 셀프 리뷰에서 발견한 이슈

1. **Quality Gate 위치 미정** — Collect 후, Load 후 검증이 필요한데 구체적 방법 미결
2. **Enrich 통합 리스크** — 회사는 Tag/TagPrep 2단계인데 1단계로 합침. 데이터 소스 3개 이상이면 문제?
3. **View vs Table** — Staging/Intermediate를 View로 할지 Table로 할지
4. **Config-driven 범위** — 어느 단계까지 설정 파일로 관리할지

## 질문

### Q1. 8단계가 과도한가?
학습 프로젝트이자 포트폴리오인데, 이 단계 구분이 적절한가? 합칠 것은? 더 나눌 것은?

### Q2. Enrich를 Tag/TagPrep으로 나눌 필요가 있나?
회사는 Excel 입력의 복잡도 때문에 2단계였다. API JSON이 주 입력인 우리도 나눠야 하나?

### Q3. DuckDB에서 dbt 패턴(staging→intermediate→mart) 구현 방법
BigQuery/Snowflake용 패턴인데 DuckDB에서도 자연스러운가? View vs Table 권장은?

### Q4. 포트폴리오 관점에서 가장 임팩트 있는 단계는?
면접관이 "이거 좀 보여주세요"라고 했을 때 가장 인상적인 부분은 어디?

### Q5. 빠진 것이 있나?
데이터 리니지, 에러 핸들링, 멱등성 등 내가 놓치고 있는 중요한 고려사항?

## 요청
- 각 질문에 동의/반대 + 구체적 이유로 답변해줘
- "괜찮다", "좋다"로 넘어가지 말고 개선 가능한 점이 있으면 구체적으로 제시
- Anti-Simple Bias 적용 (합치자/심플하게 방향 자제)
