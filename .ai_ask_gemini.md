# Gemini에게 설계 리뷰 요청

## 컨텍스트
storage/warehouse 모듈 설계를 완료했어. DuckDB 기반 로컬 데이터 웨어하우스야.

설계 문서: `docs/decisions/004-storage-design.md`

## 리뷰 요청 사항

### 1. 전체 설계 평가
- 아키텍처가 적절한가?
- 기존 Collector 패턴과 일관성이 있는가?
- 놓친 부분이 있는가?

### 2. 구체적인 질문 (ADR에 포함됨)

1. **추상 클래스 설계**: `connect()/close()` 대신 context manager (`__enter__/__exit__`)가 더 나을까?

2. **스키마 관리**: TableSchema를 코드로 정의하는 것 vs SQL 파일로 관리하는 것?

3. **에러 핸들링**: DuckDB 예외를 그대로 노출할지, 커스텀 예외로 감쌀지?

4. **테스트 전략**: in-memory DuckDB (`:memory:`)로 테스트하면 충분할까?

### 3. 개선 제안
- 더 나은 설계가 있다면 알려줘
- 2026 Python best practices 관점에서 피드백

## 첨부: 설계 문서 핵심 부분

```python
# BaseWarehouse 인터페이스
class BaseWarehouse(ABC):
    @abstractmethod
    def connect(self) -> None: ...
    @abstractmethod
    def close(self) -> None: ...
    @abstractmethod
    def create_table(self, name: str, schema: TableSchema) -> None: ...
    @abstractmethod
    def insert(self, table: str, data: list[dict]) -> InsertResult: ...
    @abstractmethod
    def query(self, sql: str) -> QueryResult: ...
    @abstractmethod
    def load_parquet(self, table: str, path: Path) -> InsertResult: ...

# 스키마 정의
@dataclass
class TableSchema:
    name: str
    columns: list[Column]
    def to_ddl(self) -> str: ...

# 결과 객체
@dataclass
class InsertResult:
    table: str
    rows_affected: int
    status: str
    errors: list[str]

@dataclass
class QueryResult:
    columns: list[str]
    rows: list[tuple]
    def to_dataframe(self) -> pd.DataFrame: ...
```
