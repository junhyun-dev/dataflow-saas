제시해주신 프로젝트 컨텍스트와 파이프라인 설계 초안을 깊이 있게 검토했습니다. B2B SaaS 데이터 파이프라인의 복잡성을 업계 표준(Medallion + dbt)으로 일반화하려는 접근이 매우 훌륭합니다. 

요청하신 **Anti-Simple Bias(단순화 지양, 필요한 복잡성 수용)** 원칙을 엄격히 적용하여, "토이 프로젝트니까 적당히 합치자"는 식의 타협 없이 현업 시니어 데이터 엔지니어의 관점에서 구체적이고 비판적인 리뷰를 제공합니다.

### Q1. 8단계가 과도한가? (합치거나 나눌 것은?)
**[반대] 과도하지 않습니다. 오히려 B2B SaaS의 특성을 보여주기 위해 유지/강화해야 합니다.**

학습용/포트폴리오라고 해서 파이프라인을 3~4단계로 축소하면, 현업에서 겪는 '데이터 품질 문제', '스키마 변경', '다중 테넌트(Multi-tenant) 처리' 등의 복잡성을 해결할 기회를 잃게 됩니다. 8단계는 논리적으로 타당하며 다음과 같이 역할을 명확히 해야 합니다.

*   **개선 제안 (경계의 명확화):**
    *   **Step 1~3 (Memory/File Layer):** Python 엔진이 주도. 외부 데이터를 가져와서(Collect), 포맷을 맞추고(Convert), SaaS 필수 메타데이터를 부여하여(Enrich) Parquet 파일로 Object Storage(로컬 파일시스템)에 떨구는 단계.
    *   **Step 4 (Ingestion Layer):** File -> DB로 넘어가는 관문.
    *   **Step 5~8 (Compute/DB Layer):** DuckDB 엔진이 주도. SQL(또는 dbt)을 통해 데이터 웨어하우스 내부에서 변환이 일어나는 단계.
*   **추가할 단계 (Quality Gate):** 8단계를 줄일 것이 아니라, Step 2와 3 사이, Step 6과 7 사이에 **Data Contract (품질 검증)** 단계를 명시적 컴포넌트로 분리해야 합니다.

### Q2. Enrich를 Tag/TagPrep으로 나눌 필요가 있나?
**[동의] API JSON이라도 2단계로 나누는 것이 아키텍처상 유리합니다.**

Excel보다 API JSON이 깔끔할 것 같지만, 실제 API 데이터는 깊은 중첩(Nested) 구조, 페이지네이션 메타데이터 혼재, 예고 없는 스키마 변경 등 다른 종류의 복잡성을 가집니다. 회사의 Tag/TagPrep 패턴은 데이터 엔지니어링의 **"Normalization(정규화)"**과 **"Enrichment(풍부화)"**의 분리로 볼 수 있습니다.

*   **구체적 이유:**
    *   **Prep (Normalization):** JSON의 Nested 구조를 Flat하게 풀고(Unnest), 날짜 포맷을 통일하고, NULL 값을 처리하는 등 **'형태'**를 맞추는 책임.
    *   **Tag (Enrichment):** `tenant_id`(고객사 식별자), `source_system`(github vs gitlab), `ingested_at`, `batch_id` 등 SaaS 비즈니스 로직에 필요한 **'메타데이터'**를 주입하는 책임.
*   **결론:** 두 책임이 다르므로 코어 모듈을 분리하십시오. 소스가 3개 이상(예: GitHub, GitLab, Jira)으로 늘어날 때, Prep은 소스마다 다르지만 Tag(Enrich) 로직은 공통화할 수 있어 확장에 매우 유리해집니다.

### Q3. DuckDB에서 dbt 패턴 구현 시 View vs Table 권장
**[구체적 가이드] DuckDB의 특성(In-process OLAP)을 고려한 하이브리드 전략이 필요합니다.**

BigQuery/Snowflake는 분산 처리 환경이므로 View의 연산 비용이 클 수 있지만, DuckDB는 단일 노드 메모리 기반 벡터화 엔진이므로 View 처리가 극단적으로 빠릅니다. 하지만 모든 것을 View로 하면 FastAPI 서빙 시 매번 전체 파이프라인이 재계산되는 병목이 생깁니다.

*   **Staging (Silver): 무조건 `VIEW`**
    *   단순 컬럼명 변경, 타입 캐스팅만 수행하므로 DuckDB에서는 Parquet 파일 위에 View를 씌우는 것(`CREATE VIEW AS SELECT * FROM read_parquet(...)`)이 가장 효율적입니다.
*   **Intermediate (Silver): `VIEW` 또는 `TABLE` (조건부)**
    *   단순 JOIN이나 필터링이라면 `VIEW`.
    *   복잡한 Window 함수 연산, 대규모 Group By가 포함된다면 `TABLE` (또는 DuckDB의 임시 테이블).
*   **Mart / Report (Gold): 무조건 `TABLE` (Materialized)**
    *   FastAPI가 실시간으로 찌르는 엔드포인트입니다. API 응답 속도를 위해 반드시 물리적 테이블(Parquet 백업 기반)로 적재되어야 합니다.

### Q4. 포트폴리오 관점에서 가장 임팩트 있는 단계는?
**면접관을 사로잡을 핵심은 "Step 4 (Load)의 멱등성 보장"과 "Config-driven Architecture"입니다.**

데이터 엔지니어 면접에서 가장 집요하게 파고드는 질문은 *"DAG가 중간에 실패해서 재시도했을 때 데이터가 중복되나요?"*와 *"새로운 소스를 추가하려면 코드를 얼마나 짜야 하나요?"* 입니다.

1.  **멱등성(Idempotency)이 보장된 Load (Upsert/Merge 패턴):**
    *   단순 `INSERT INTO`가 아니라, `batch_id`나 `updated_at`을 기준으로 기존 데이터를 덮어쓰거나(Merge), 파티션을 교체(Partition Swap)하는 로직을 DuckDB로 구현해 보세요.
2.  **Config-driven (설정 기반) 파이프라인 (Step 1~3):**
    *   Python 코드를 수정하지 않고, YAML이나 JSON 설정 파일(Pydantic으로 검증)만 추가하면 새로운 GitHub API 엔드포인트(예: PR, Issues, Commits) 수집부터 Parquet 변환까지 자동으로 DAG가 생성되는 구조를 보여주세요.

### Q5. 빠진 것이 있나? (놓치고 있는 중요한 고려사항)
설계는 훌륭하나, B2B SaaS라는 도메인 특성을 고려할 때 다음 3가지가 누락되어 있습니다.

1.  **다중 테넌트(Multi-Tenancy) 데이터 격리 전략**
    *   SaaS의 핵심은 A 고객사가 B 고객사의 데이터를 보지 못하게 하는 것입니다.
    *   **개선점:** Storage 경로를 `s3://bucket/tenant_id=xxx/source=github/...` 형태로 파티셔닝하고, DuckDB Load 시 Row-level security 또는 테넌트별 View를 생성하는 로직이 파이프라인에 포함되어야 합니다.
2.  **Data Contract (Quality Gate)**
    *   셀프 리뷰에서 고민하셨던 부분입니다. 파이프라인 중간에 데이터 품질을 검증하는 단계가 명시적으로 있어야 합니다.
    *   **개선점:** Collect 후(JSON 스키마 검증), Load 전(Null 비율, PK 중복 검증)에 Pydantic이나 간단한 SQL assert 문을 실행하여, 실패 시 파이프라인을 중단(Circuit Breaker)시키는 로직을 추가하세요.
3.  **스키마 진화 (Schema Evolution) 대응**
    *   GitHub API가 갑자기 응답에 새로운 필드를 추가하거나, 기존 필드 타입을 바꾸면 어떻게 되나요?
    *   **개선점:** DuckDB의 `UNION BY NAME` 기능이나, Parquet 생성 시 Schema Merge 옵션을 활용하여 파이프라인이 깨지지 않고 유연하게 대응하는 처리 로직을 추가하면 매우 높은 평가를 받을 수 있습니다.
