# ADR 001: 프로젝트 구조 - SaaS 스타일 모노레포

## 상태

승인됨 (2026-03-02)

## 컨텍스트

회사에서 배운 데이터 엔지니어링 패턴을 정리하고 학습하기 위한 토이 프로젝트가 필요했다.

**고려한 옵션들**:

1. **여러 개의 작은 레포지토리**
   - 장점: 각 컴포넌트 독립적
   - 단점: 통합 어려움, 관리 복잡

2. **단일 모노레포 (선택)**
   - 장점: 통합 쉬움, 전체 그림 파악 용이
   - 단점: 규모가 커지면 복잡

3. **회사 구조 그대로 복사**
   - 장점: 회사 코드와 1:1 매핑
   - 단점: 독점 코드 문제, 학습보다 복사에 가까움

## 결정

**SaaS 스타일 모노레포**를 선택했다.

```
dataflow-saas/
├── auth/           # 인증
├── ingestion/      # 수집
├── transform/      # 변환
├── storage/        # 저장
├── analytics/      # 분석
├── visualization/  # 시각화
├── orchestration/  # 오케스트레이션
├── observability/  # 모니터링
├── agents/         # AI 에이전트
└── docs/           # 문서
```

## 근거

1. **학습 목적에 맞음**
   - 전체 SaaS 아키텍처를 한 곳에서 볼 수 있음
   - 컴포넌트 간 관계 이해 용이

2. **포트폴리오로 적합**
   - 면접에서 "전체 시스템 설계 경험" 어필 가능
   - 특정 영역만 보여줄 수도 있음

3. **회사 경험 반영**
   - 회사 hyperloungescripts의 구조 참고
   - 하지만 독립적으로 새로 설계

4. **확장성**
   - GitHub 데이터뿐 아니라 다른 도메인으로 확장 가능
   - 새 컴포넌트 추가 쉬움

## 결과

- 프로젝트 구조 생성 완료
- 각 디렉토리에 독립적인 requirements.txt
- 공통 문서는 docs/에 집중
